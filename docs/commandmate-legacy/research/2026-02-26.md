## 1. Chủ đề

**"CLI vs MCP: The Hidden Tax on AI Agents & Tool Access Optimization"**

Khi xây dựng AI agents, hầu hết người ta tập trung vào model (GPT-4 vs Claude), orchestration (multi-agent vs single-agent), hoặc prompts. Nhưng có một vấn đề ẩn, **tốn kém im lặng**, ít ai chú ý: **token cost của tool discovery**.

**Vấn đề:**
- Mỗi khi agent khởi động session → cần biết "tôi có quyền làm gì?" → load tool catalog
- **MCP (Model Context Protocol)** — chuẩn công nghiệp hiện tại — dump TOÀN BỘ tool schema vào context (JSON Schema đầy đủ cho mọi tool)
- **CLI (Command-Line Interface)** — cách cũ hơn — chỉ load tool names, discover details on-demand

**Kết quả:**
> MCP tốn **~15,540 tokens** mỗi session start (84 tools).  
> CLI tốn **~300 tokens** session start, ~910 tokens cho 1 tool call.  
> **Tiết kiệm 94% tokens** khi sử dụng CLI thay vì MCP (cho use case thông thường).

**Tại sao quan trọng:**
1. **Cost:** Token costs = real money. $10-50/tháng → $100-500/tháng nếu scale
2. **Latency:** Ít tokens → faster session start, faster tool calls
3. **Model limits:** Context window là tài nguyên khan hiếm (8K - 128K tokens)
4. **Sustainability:** Outcome-based pricing cần tối ưu cost để competitive

**Nguồn:** [Kan Yilmaz — "I Made MCP 94% Cheaper (And It Only Took One Command)"](https://kanyilmaz.me/2026/02/23/cli-vs-mcp.html) + ["Coding Agents Are at Stage 5, Everything Else Is Stuck at Stage 1"](https://kanyilmaz.me/2026/02/19/five-stages-of-ai-agents.html)

## 2. Tóm tắt: The Hidden Tax of MCP

### A. How MCP Works (Current Industry Standard)

**MCP = Model Context Protocol** — được Anthropic giới thiệu (2024), trở thành chuẩn de-facto cho agent tool access.

**Workflow:**

1. **Session start:**
   - Agent connects to MCP server
   - MCP server returns **full tool catalog** (JSON Schema cho mọi tool)
   - Example:
     ```json
     [
       {
         "name": "notion-search",
         "description": "Search for pages and databases",
         "inputSchema": {
           "type": "object",
           "properties": {
             "query": { "type": "string", "description": "The search query text" },
             "filter": {
               "type": "object",
               "properties": {
                 "property": { "type": "string", "enum": ["object"] },
                 "value": { "type": "string", "enum": ["page", "database"] }
               }
             }
           }
         }
       },
       {
         "name": "notion-create-page",
         "description": "Create a new page",
         ...
       },
       ... (84 tools total)
     ]
     ```
   - **Token cost:** ~185 tokens per tool × 84 tools = **15,540 tokens**

2. **Tool call:**
   - Agent decides to use tool → calls MCP API
   - Example:
     ```json
     {
       "tool_call": {
         "name": "notion-search",
         "arguments": { "query": "my search" }
       }
     }
     ```
   - **Token cost:** ~30 tokens per call (cheap because schema already loaded)

**Total cost for typical session (1 tool call):**
- Session start: 15,540 tokens
- 1 tool call: 30 tokens
- **Total: 15,570 tokens**

**For 10 tool calls:**
- Session start: 15,540 tokens
- 10 tool calls: 300 tokens
- **Total: 15,840 tokens**

### B. How CLI Works (Old School, New Relevance)

**CLI = Command-Line Interface** — classic Unix approach: discover tools when you need them.

**Workflow:**

1. **Session start:**
   - Agent loads **lightweight skill listing** (just names + locations)
   - Example (OpenClaw's `available_skills` format):
     ```xml
     <available_tools>
       <tool>
         <name>notion</name>
         <description>CLI for Notion</description>
         <location>~/bin/notion</location>
       </tool>
       <tool>
         <name>linear</name>
         <description>CLI for Linear</description>
         <location>~/bin/linear</location>
       </tool>
       ... (6 tools total)
     </available_tools>
     ```
   - **Token cost:** ~50 tokens per tool × 6 tools = **300 tokens**

2. **Tool call (two-step):**

   **Step 1: Discovery** (~610 tokens)
   ```bash
   $ notion --help
   
   notion search <query> [--filter-property ...]
     Search for pages and databases
   
   notion create-page <title> [--parent-id ID]
     Create a new page
   
   notion update-page <page-id> [--title TITLE] [--content CONTENT]
     Update an existing page
   
   ... (12 more tools)
   ```
   - **Token cost:** ~600 tokens (full CLI reference for 14 commands)

   **Step 2: Execute** (~6 tokens)
   ```bash
   $ notion search "my search"
   ```
   - **Token cost:** ~6 tokens

**Total cost for typical session (1 tool call):**
- Session start: 300 tokens
- Discovery (1 tool): 600 tokens
- Execute (1 call): 10 tokens
- **Total: 910 tokens**

**For 10 tool calls (assume 1 discovery covers all):**
- Session start: 300 tokens
- Discovery (1 tool): 600 tokens
- Execute (10 calls): 60 tokens
- **Total: 964 tokens**

### C. Token Cost Comparison

| Tools Used | MCP | CLI | Savings |
|------------|-----|-----|---------|
| **Session start** | ~15,540 | ~300 | **98%** |
| **1 tool** | ~15,570 | ~910 | **94%** |
| **10 tools** | ~15,840 | ~964 | **94%** |
| **100 tools** | ~18,540 | ~1,504 | **92%** |

**Key insight:**
> CLI uses **~94% fewer tokens** overall for typical workflows (1-10 tool calls).

**Why CLI wins:**
1. **Lazy loading:** Only loads tool details when needed (not upfront)
2. **Compressed format:** CLI help text is denser than verbose JSON Schema
3. **Shared discovery:** One `--help` call covers all commands for that CLI

**Why MCP still popular:**
- Standardized (Anthropic-backed, industry adoption)
- Easier for LLM to parse (structured JSON vs unstructured text)
- Better DX for tool developers (schema-first design)

### D. Anthropic's Tool Search (The Middle Ground)

**Problem:** Anthropic recognized MCP's token waste.

**Solution:** [Tool Search](https://www.anthropic.com/engineering/advanced-tool-use) — load a **search index** instead of full schemas, fetch on-demand.

**Workflow:**
1. **Session start:** Load lightweight index (~500 tokens)
2. **Tool discovery:** Agent searches for relevant tools → fetch 3-5 tools (~3K tokens)
3. **Tool call:** Standard MCP call (~30 tokens)

**Token cost comparison:**

| Tools Used | MCP | Tool Search | CLI | Savings (CLI vs TS) |
|------------|-----|-------------|-----|---------------------|
| **Session start** | ~15,540 | ~500 | ~300 | **40%** |
| **1 tool** | ~15,570 | ~3,530 | ~910 | **74%** |
| **10 tools** | ~15,840 | ~3,800 | ~964 | **75%** |
| **100 tools** | ~18,540 | ~12,500 | ~1,504 | **88%** |

**Key insight:**
- Tool Search is better than MCP (85% savings for 1 tool), but **CLI is still 74% cheaper than Tool Search**
- Tool Search is **Anthropic-only** (vendor lock-in), CLI works with any model
- Tool Search still fetches full JSON Schema per tool (not as compact as CLI help text)

**When to use each:**

| Approach | Best For |
|----------|----------|
| **MCP** | Workflows where agent uses MOST tools (50%+ of catalog) |
| **Tool Search** | Anthropic users, workflows with 5-20 tools used per session |
| **CLI** | Cost-sensitive, multi-model, workflows with 1-10 tools used per session |

## 3. Liên quan CommandMate

### Trụ 1: Outcome-Based Pricing

**Tool access cost = hidden tax on pricing.**

**Scenario without optimization:**
- CommandMate sử dụng MCP (industry standard)
- Mỗi task = 1 agent session = 15,540 tokens session start
- 1,000 tasks/tháng = 15.54M tokens/tháng cho session starts alone
- Token cost (Claude Opus): $15/M input tokens → **$233/tháng chỉ để load tools**
- Chưa kể tokens cho actual work (prompts, outputs, tool calls)

**With CLI optimization:**
- Session start: 300 tokens (thay vì 15,540)
- 1,000 tasks/tháng = 0.3M tokens/tháng
- Token cost: $4.50/tháng
- **Tiết kiệm $228/tháng** (98% savings)

**Impact on pricing:**

**Without optimization:**
- Cost per task: $0.233 (session start) + $0.50 (work) = **$0.733**
- Để profitable (40% margin) → phải charge **$1.22/task**
- Customer perception: "Expensive! Competitor charges $0.80/task"

**With optimization:**
- Cost per task: $0.005 (session start) + $0.50 (work) = **$0.505**
- Để profitable (40% margin) → charge **$0.84/task**
- **31% cheaper than competitor** → win deals

**Lesson:**
> "Outcome-based pricing requires ruthless cost optimization. MCP's convenience is a $200+/month tax at scale."

**Key insight:**

**CommandMate's vision:**
- Support 100+ tools (Notion, Linear, GitHub, Slack, Gmail, Calendar, Stripe, Vercel, etc.)
- Agents tự chọn tools cần thiết

**CLI approach:**
- 100 tools = $0.005/task session start (just tool names)

**Conclusion:**
> "CLI enables CommandMate's 'tool library' model without breaking the bank."

**What to track:**

### Phase 3 (Post-Launch): Advanced Optimizations

#### D. Intelligent Caching (Discovery Results)

**Problem:** Agent discovers `notion --help` multiple times → redundant tokens.

**Solution:** Cache discovery results per tool version.

**Implementation:**

1. **Cache schema:**
   ```json
   {
     "tool": "notion",
     "version": "1.2.0",
     "help_text": "notion search <query> [--filter-property ...]\n...",
     "cached_at": "2026-02-26T08:00:00Z",
     "ttl_hours": 24
   }
   ```

2. **Workflow:**
   - Agent runs `notion --help`
   - CommandMate checks cache:
     - **Cache hit:** Return cached text (0 tokens for discovery)
     - **Cache miss:** Execute `--help`, cache result (600 tokens)
   - TTL: 24 hours (invalidate daily to catch updates)

**Savings:**
- First discovery: 600 tokens
- Subsequent discoveries (within 24h): 0 tokens
- Average: 50-100 tokens/discovery (amortized)

**Effort:** 2 weeks (caching layer, invalidation logic)

## 5. The 5 Stages of AI Agents (Tool Access Perspective)

**Context:** Kan Yilmaz published ["Coding Agents Are at Stage 5, Everything Else Is Stuck at Stage 1"](https://kanyilmaz.me/2026/02/19/five-stages-of-ai-agents.html)

**5 Stages of Agent Maturity:**

| Stage | Name | What It Means | Examples |
|-------|------|---------------|----------|
| **1** | Tool Access | Agent can read, write, execute everything it needs | Bash, browser, CMS write access |
| **2** | Planning | Breaks work into steps, tackles sequentially | Task lists, dependency ordering |
| **3** | Verification | Tests its own output, catches mistakes, iterates | Running tests, confirming actions |
| **4** | Personalization | Follows your conventions, style, constraints | SOUL.md, SKILL.md, AGENTS.md |
| **5** | Memory & Orchestration | Manages context, delegates to sub-agents, parallel work | Planner, Orchestrator, Worker, QA agents |

**Key insight:**
> "Coding agents are at Stage 5. Everything else (SEO, Sales, Marketing, Ops) is stuck at Stage 1."

**Why?**
- **Coding agents:** Bash = universal tool, one interface to everything (file system, git, package managers, testing)
- **Other agents:** Need dozens of specialized tools (GA4, Search Console, Ahrefs, CRM, email, calendar, ad platforms, etc.)

**The Tool Access Barrier:**

| Workflow | What Coding Agents Have | What Others Need |
|----------|-------------------------|------------------|
| **Coding** | Bash, file read/write | *(Solved)* |
| **SEO** | — | GA4, Search Console, Ahrefs, CMS write, Lighthouse, G2, social accounts, wiki, YouTube, email... |
| **Sales** | — | CRM read/write, email send, calendar, enrichment APIs, verification, analytics... |
| **Marketing** | — | Ad platforms, analytics, social scheduling, design tools, video tools, social accounts... |

**Lesson for CommandMate:**
> "Most AI agent startups focus on Stages 2-5 (planning, orchestration, memory). **Stage 1 (tool access) is the bottleneck.** Solve it FIRST, unlock value immediately."

**CommandMate's opportunity:**
- Be the platform that solves Stage 1 for EVERY workflow (not just coding)
- CLI + MCP + API aggregation = comprehensive tool access
- "Coding agents had bash. Your agents have CommandMate."

## 7. Risks & Mitigations

### Risk 1: CLI Help Text Inconsistency

**Risk:** CLI help text format varies (some verbose, some cryptic) → LLM struggles to parse.

**Example:**
- Good: `notion search <query> [--filter-property PROP] [--filter-value VAL]`
- Bad: `notion search [options] <args>` (no detail)

**Mitigation:**
- Standardize help text format (require detailed descriptions)
- Auto-generate help from OpenAPI specs (consistent structure)
- Fallback to MCP if CLI help is poor quality

---

### Risk 2: Discovery Latency

**Risk:** `--help` call = subprocess execution = 100-500ms latency per discovery.

**Mitigation:**
- Cache discovery results (24h TTL) → 0ms for subsequent calls
- Pre-load popular tools for power users
- Parallel discovery (if agent needs multiple tools, fetch all `--help` concurrently)

---

### Risk 3: CLI Maintenance Burden

**Risk:** 100 CLIs = 100 tools to maintain (updates, bug fixes, auth changes).

**Mitigation:**
- Auto-generate CLIs from OpenAPI specs (upstream changes propagate automatically)
- Leverage community (CLIHub directory, accept contributions)
- Prioritize top 20 tools (80/20 rule: 20% of tools = 80% of usage)

---

### Risk 4: LLM Struggles with CLI Output Parsing

**Risk:** CLI output is unstructured text (vs JSON from MCP) → LLM misinterprets.

**Example:**
```bash
$ notion search "my query"
Found 3 pages:
1. Project Plan (page_abc123)
2. Meeting Notes (page_def456)
3. Roadmap (page_ghi789)
```

**LLM might:**
- Miss page IDs
- Confuse titles
- Fail to extract structured data

**Mitigation:**
- **CLI output format:** Always return JSON when possible
  ```bash
  $ notion search "my query" --format json
  [
    {"id": "page_abc123", "title": "Project Plan"},
    {"id": "page_def456", "title": "Meeting Notes"},
    {"id": "page_ghi789", "title": "Roadmap"}
  ]
  ```

## 8. Key Lessons for CommandMate

### Lesson 1: "Token efficiency = competitive advantage"

**Insight:** Outcome-based pricing requires ruthless cost optimization.

**Math:**
- Competitor (MCP): $1.22/task (to maintain margin)
- CommandMate (CLI): $0.84/task
- **31% cheaper** → win price-sensitive customers

---

### Lesson 2: "Tool access is the bottleneck for non-coding agents"

**Insight:** Most agent platforms focus on orchestration (Stages 2-5). Stage 1 (tool access) is unsolved.

**CommandMate opportunity:**
- Build comprehensive tool library (100+ tools)
- Solve auth, rate limits, error handling
- "You bring the task, we bring the tools"

---

### Lesson 3: "CLI vs MCP is not either/or — it's both"

**Insight:** Hybrid approach wins.

**Strategy:**
- CLI for core tools (20 tools, 80% of usage) → max savings
- MCP for long-tail tools (80 tools, 20% of usage) → ease of integration
- Data-driven switching (if tool usage spikes → pre-load)

---

### Lesson 4: "Lazy loading > eager loading"

**Insight:** Don't load what you don't need.

**Application:**
- Session start: Load tool names only (300 tokens)
- Tool usage: Discover on-demand (600 tokens first time, 0 tokens cached)
- Amortized cost: ~50 tokens/tool (vs 185 tokens/tool for MCP upfront)

---

### Lesson 5: "Agent tool usage patterns are predictable"

**Insight:** 80% of tasks use 20% of tools.

**Data to collect:**
- Task type → tool usage (e.g., "Deploy" tasks always use `vercel`, `github`)
- Agent role → tool usage (e.g., Kiến uses `vercel`, Minh uses `notion`)

**Optimization:**
- Pre-load predictable tools
- Lazy-load rare tools
- Fine-tune per agent/task type
