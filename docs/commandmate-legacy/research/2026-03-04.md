# Nghiên cứu: Agent Unit Economics 2.0 — Từ giá theo token sang giá theo runtime + state infrastructure (2026-03-04)

## 1) Chủ đề A: OpenAI chuyển container billing sang mô hình thời gian phiên (20 phút) — tín hiệu mạnh cho “runtime economics”

### Tóm tắt
- OpenAI thông báo từ **31/03/2026**, container usage (Hosted Shell/Code Interpreter) sẽ tính theo **phiên 20 phút/container** thay vì đơn vị container flat như trước.
- Bảng giá tools cho thấy cost stack của agent giờ gồm nhiều lớp: **token + tool call + storage + runtime session**.
- Giá tool call đã rõ hơn: web search theo 1k calls, file search theo 1k calls + storage GB/day.
- Điều này làm lộ rõ thực tế: với agent workflow phức tạp, chi phí không còn chủ yếu ở model token mà nằm ở **orchestration runtime** và số lần gọi tool.

### Liên quan CommandMate
- **Trụ 1 — Outcome pricing:** Nếu CommandMate chỉ quote theo token hoặc theo task count, biên lợi nhuận dễ bị bào mòn khi agent “chạy lâu” trong shell/container.
- **Trụ 2 — Tools cho agents:** Cần đo usage theo tầng tool (calls, duration, retries) để tránh hidden cost.
- **Trụ 3 — System of record:** SoR phải lưu được “cost provenance” theo từng run để giải thích tại sao 1 outcome tốn khác outcome khác.

### Đề xuất cụ thể cho CommandMate
1. **Thêm Runtime Budget Guardrail**
   - Mỗi workflow có ngân sách theo phút runtime + số lần tool call.
   - Khi vượt ngưỡng, agent tự chuyển mode: “summarize + ask user approval” thay vì tiếp tục tự chạy.
2. **Outcome Cost Ledger v1**
   - Mỗi outcome ghi 4 cột chuẩn: token cost, tool-call cost, runtime-session cost, retry cost.
   - Dùng ledger này để thiết kế gói giá theo “resolved outcome with cap”.
3. **Pricing chính thức theo 2-part tariff**
   - Phần A: phí outcome cố định.
   - Phần B: surcharge minh bạch nếu vượt budget runtime (do user yêu cầu thêm hoặc data quality thấp).

## 2) Chủ đề B: LangSmith đang productize “agent cloud” với đơn vị giá theo trace/run/uptime — SoR dần trở thành sản phẩm có giá riêng

### Tóm tắt
- Trang pricing của LangSmith cho thấy cấu trúc giá không chỉ seat: có **trace volume**, **deployment run**, **uptime cost theo phút**, và quota Agent Builder runs.
- Ngoài observability/evals, họ nhấn mạnh deployment có **state/memory APIs**, cron scheduling, authz, và khả năng expose agent thành MCP server.
- Đây là tín hiệu thị trường: “hệ thống ghi nhận hoạt động agent” (trace + state + memory) đang chuyển từ phụ trợ kỹ thuật sang **business-critical infrastructure** có đơn giá rõ.

### Liên quan CommandMate
- **Trụ 1 — Outcome pricing:** Khi thị trường quen với model pay-per-run + pay-per-uptime, CommandMate có thể định giá outcome nhưng nội bộ phải map ngược về run economics.
- **Trụ 2 — Tools cho agents:** Tool platform không chỉ là connector; phải đi kèm deployment reliability và monitoring để giữ quality outcome.
- **Trụ 3 — System of record:** SoR của CommandMate nên được định nghĩa như “control plane” (state, memory, trace, approval, audit), không chỉ là chỗ lưu log.

### Đề xuất cụ thể cho CommandMate
1. **SoR as Product Surface (không chỉ internal log):**
   - Mở dashboard cho khách xem: timeline run, tool usage, approval checkpoints, cost-to-resolution.
2. **Resolution Evidence Pack:**
   - Mỗi outcome xuất kèm “bằng chứng hoàn thành” (data snapshot + trace summary + ai/nguời quyết định ở điểm nào).
3. **SKU mới: Managed Agent Reliability Add-on**
   - Thu phí add-on cho SLA, monitoring, replay/debug, và governance reports.
   - Giúp tăng ARPU mà không phá cấu trúc outcome pricing chính.

## Kết luận nhanh cho hôm nay
- Tín hiệu mới đáng chú ý: thị trường agent đang dịch từ “model pricing” sang “**execution economics + reliability economics**”.
- Cửa thắng cho CommandMate: giữ mặt ngoài là **outcome pricing dễ hiểu cho SME**, nhưng mặt trong phải có **cost ledger + runtime governance + SoR có khả năng giải trình**.
- Nếu làm tốt, CommandMate không chỉ là “tool chạy agent” mà là “hệ thống bảo chứng kết quả có kiểm soát chi phí”.
